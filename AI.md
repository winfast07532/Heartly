# CLAUDE.md — HEARTLY PROJECT CONTEXT

## PROJECT NAME
Heartly

Heartly is a modern emotional relationship/love app focused on:
- emotional connection
- immersive UI/UX
- memories
- messaging
- streak systems
- personalization
- premium visual feel

The app should feel emotionally polished, modern, smooth, and premium.

---

# CORE DEVELOPMENT RULES

- Do NOT break existing features
- Do NOT rewrite working code unless explicitly asked
- Preserve current architecture and styling consistency
- Always integrate new features into the current system properly
- Avoid isolated or disconnected features

---

# PRIMARY TECH STACK

ONLY USE:
- HTML
- CSS
- JavaScript

DO NOT:
- add frameworks
- introduce React/Vue/etc
- restructure into large frameworks
- introduce unnecessary dependencies

Keep the project lightweight and maintainable.

---

# DESIGN SYSTEM

Heartly should feel:
- emotional
- modern
- cinematic
- soft
- premium
- intimate
- smooth

UI priorities:
- clean spacing
- modern typography
- smooth transitions
- polished animations
- responsive layout
- minimal clutter

Avoid:
- generic layouts
- ugly spacing
- inconsistent sizing
- random colors
- outdated UI patterns
- overcomplicated visuals

No emojis unless explicitly requested.

Use:
- icons
- subtle glow
- blur
- gradients
- depth
- smooth hover states

---

# UI CONSISTENCY RULES

Every addition must match:
- existing color palette
- spacing system
- border radius style
- animation timing
- typography hierarchy
- interaction behavior

Do NOT create visually disconnected components.

---

# FEATURE INTEGRATION RULES

All features must integrate naturally with:
- messages
- memories
- streaks
- profiles
- relationship systems
- existing navigation
- current storage systems

Avoid:
- demo-style features
- fake placeholders
- isolated widgets

Everything should feel like part of one cohesive product.

---

# CODE STYLE RULES

Code should be:
- clean
- readable
- modular
- scalable
- efficient

Avoid:
- bloated files
- duplicate logic
- unnecessary complexity
- random inline styling everywhere
- breaking existing naming conventions

Prefer:
- reusable structures
- reusable utility classes
- maintainable organization

---

# PERFORMANCE PRIORITIES

Heartly should feel:
- smooth
- responsive
- lightweight
- fluid

Optimize:
- animations
- rendering
- transitions
- DOM updates

Avoid:
- unnecessary heavy effects
- laggy animations
- excessive event listeners
- memory leaks

---

# OUTPUT RULES

When generating code:
- provide FULL working code
- preserve compatibility with existing codebase
- modify only relevant sections
- do not rewrite entire files unless requested

No explanations unless requested.

---

# BEHAVIOR RULES

Before changing anything:
- understand current structure
- preserve existing systems
- maintain consistency

If something could break existing functionality:
- mention it first

Do not assume this is a throwaway prototype.

---

# DEVELOPMENT GOAL

Heartly should feel like:
- a polished real product
- emotionally engaging
- premium quality
- cohesive and immersive

NOT:
- a random collection of features
- a generic template
- a messy experimental app

Every change should improve:
- UX
- emotional feel
- polish
- consistency
- responsiveness
